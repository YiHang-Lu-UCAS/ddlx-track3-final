# DDL-X Track 3 Compliance Audit

This document maps the verification bundle to the Track 3 requirements and clarifies which parts are model-generated.

## Official requirements checked

From the Track 3 rules, winning solutions should provide:

1. Complete implementation.
2. Training pipelines and hyperparameter configurations.
3. Evaluation code and reproducibility documentation.
4. Final model weights in standard formats.
5. Only organizer-recommended/public released datasets for training/evaluation.
6. Any generation tools used for extended samples must be submitted for reproducibility.

## Are all submitted fields model-generated?

### `Classification result`

Generated by the ConvNeXt-B face-level classifier:

```text
models/classifier/convnextb_cls_dev_adapt_head_stage4_last.pt
```

The pipeline aggregates face scores with `max(face_fake_probability)` and uses threshold `0.20`. The final WBF packaging reused the generated `image_scores.csv`; it did not manually assign labels.

### `Bounding boxes`

Generated by detector models plus deterministic post-processing:

```text
models/detectors/old_fullmask_continue96_stageb3_best.pt
models/detectors/repeat2_conservative_lr1e4_stageb_best.pt
models/detectors/yolov8m512_stageab_stageb_best.pt
src/ddli_detector_v1/merge_wbf_test_boxes_hetero_v1.py
```

The final method uses WBF with fixed parameters in `configs/final_wbf.yaml`.

For fake images where no WBF detector box was retained, the selected final variant uses deterministic nose/eyes/mouth fallback boxes from model-derived face landmarks. These boxes are not manually drawn; they are generated by code from face landmark detections. This is post-processing, not human annotation.

### `Visible forgery traces`

Generated by Qwen2.5-VL-3B-Instruct plus LoRA checkpoint-1500:

```text
models/explanation/qwen2_5_vl_3b_instruct/
models/explanation/qwen2_5_vl_3b_lora_checkpoint1500/
scripts/run_explanation_inference.sh
scripts/run_fake_nobox_explanation_rerun.sh
```

The scripts construct prompts from the image, predicted label, and predicted boxes, then call `swift infer`. The cached zips in `text_sources/` are included only to rebuild the exact leaderboard artifact and verify hashes. They are not claimed as the method.

## Important caveat about exact final zip vs method-level regeneration

The final leaderboard zip can be exactly rebuilt from cached Qwen-generated text zips and saved WBF boxes. Re-running Qwen may produce semantically equivalent but byte-different text if library versions, CUDA kernels, or preprocessing differ. Therefore:

- For exact leaderboard artifact verification, use `scripts/rebuild_from_saved_wbf_boxes.sh` and `text_sources/`.
- For method-level explanation verification, use `scripts/run_explanation_inference.sh` or `scripts/run_fake_nobox_explanation_rerun.sh`.

## Recommended wording to organizers

The submitted system is a single Track 3 pipeline consisting of a classifier, a localization ensemble with WBF, and a Qwen-based explanation generator. All output fields were produced by trained models and deterministic post-processing; no submitted explanations or boxes were manually authored. The final package includes both exact-result verification artifacts and method-level reproduction scripts.


Detailed Qwen prompt construction, token limits, decoding settings, and text post-processing are documented in `docs/explanation_generation_details.md`.


For explanation reproduction, `QWEN_MAX_NEW_TOKENS=2048` is used by default because it was the largest verified Qwen generation setting in the progressive rerun workflow.
