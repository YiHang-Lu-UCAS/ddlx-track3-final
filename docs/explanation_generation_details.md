# Explanation Generation Details

This document describes how the `Visible forgery traces` text is produced and which generation parameters are used for reproducibility.

## Summary

The text is generated by a vision-language model, not manually written:

```text
Qwen2.5-VL-3B-Instruct base model
+ LoRA adapter checkpoint-1500
+ image
+ predicted classification label
+ predicted/fallback bounding boxes
-> Visible forgery traces
```

Bundle paths:

```text
models/explanation/qwen2_5_vl_3b_instruct/
models/explanation/qwen2_5_vl_3b_lora_checkpoint1500/
scripts/run_explanation_inference.sh
scripts/run_fake_nobox_explanation_rerun.sh
src/ddli_explain_v1/build_test_explain_shards.py
src/ddli_explain_v1/build_fake_nobox_variant_explain_shards.py
```

## Prompt construction

For a predicted fake image, the prompt is generated by code as:

```text
You are writing the Visible forgery traces field for a forensic image submission. The upstream classifier labels this image as fake. Candidate manipulated bounding boxes from the detector are: <boxes>. Describe only visually supportable manipulation traces in English, focusing on the candidate regions when present. Include a concise final statement that the image has been tampered with. Do not output JSON or bounding box coordinates.
```

If no reliable box is retained, `<boxes>` is replaced with:

```text
no reliable detector box was retained
```

For a predicted real image, the prompt is:

```text
You are writing the Visible forgery traces field for a forensic image submission. The upstream classifier labels this image as real and no manipulation box is retained. Describe visually supportable evidence of consistency in English and end with a concise statement that the image has not been tampered with. Do not output JSON or bounding box coordinates.
```

## Method-level generation parameters

The method-level reproduction scripts call `swift infer` with the bundled Qwen model and LoRA adapter:

```bash
QWEN_MAX_NEW_TOKENS=2048 bash scripts/run_explanation_inference.sh <json_dir> <test_images.csv> <out_root>
```

Internally this uses:

```bash
swift infer \
  --model models/explanation/qwen2_5_vl_3b_instruct \
  --adapters models/explanation/qwen2_5_vl_3b_lora_checkpoint1500 \
  --template qwen2_5_vl \
  --val_dataset <input_jsonl> \
  --infer_backend pt \
  --torch_dtype float16 \
  --max_length 4096 \
  --max_pixels 602112 \
  --max_new_tokens "$QWEN_MAX_NEW_TOKENS" \
  --temperature 0 \
  --num_beams 1 \
  --max_batch_size 128 \
  --stream false
```

Default reproduction parameters:

| Parameter | Default value | Meaning |
|---|---:|---|
| `QWEN_MAX_NEW_TOKENS` / `max_new_tokens` | `2048` | Largest verified generation setting from the progressive rerun workflow; used for method-level verification. |
| `max_length` | `4096` | Model preprocessing sequence setting used by Swift. |
| `max_pixels` | `602112` | Image preprocessing setting for Qwen-VL. |
| `temperature` | `0` | Deterministic decoding. |
| `num_beams` | `1` | Greedy decoding, no beam search. |
| `max_batch_size` | `128`, fallback `64` | Generation batch setting. |
| `torch_dtype` | `float16` | Inference dtype. |

`QWEN_MAX_NEW_TOKENS` can be changed by the verifier without editing code, for example:

```bash
QWEN_MAX_NEW_TOKENS=1024 bash scripts/run_explanation_inference.sh ...
QWEN_MAX_NEW_TOKENS=2048 bash scripts/run_explanation_inference.sh ...
```

## Historical final artifact parameter

The exact leaderboard artifact was generated from cached Qwen outputs produced earlier in the pipeline. The text workflow was progressive: the main full-test generation used:

```text
max_new_tokens = 640
```

Then later rerun/repair scripts used larger generation settings:

```text
launch_missing_statement_rerun_1024_6gpu_v1.sh     -> max_new_tokens=1024
launch_remaining_statement_rerun_2048_6gpu_v1.sh   -> max_new_tokens=2048
```

For method-level verification, the new reproduction entrypoints default to `2048`, the largest setting used in that progressive workflow, so the verifier can regenerate full explanations directly from the bundled model. For exact byte-level leaderboard hash verification, use the cached Qwen-generated text zips in `text_sources/`.

## Post-processing

The Qwen response replaces only the `Visible forgery traces` field. The code checks for empty responses and verifies the expected final statement:

- fake: `This image has been tampered with.`
- real: `This image has not been tampered with.`

For fake-nobox text reruns, if a generated fake response lacks the expected final statement, the postprocessor appends:

```text
Summary: This image has been tampered with.
```

This is deterministic formatting repair, not manual writing.
